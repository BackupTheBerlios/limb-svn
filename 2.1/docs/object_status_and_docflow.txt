       С любым объектов LIMB связан его статус. Каждый объект может трактовать значение этого 
поля по-своему, поэтому признак статуса может быть использован для реализации чего угодно, в 
том числе и документооборота. Все операции со статусом стоит выполнять с использованием 
битовых масок. В настоящее время мы использует поле статус для некоторых объектов для того, 
чтобы хранить признак опубликованности объекта. В одном проекте для некоторых объектов 
статус использовался для пометки объектов как особые, чтобы не вводить лишнее поле в таблицу.
       Поле статус, а также политика доступа, используемая в LIMB позволяет, как мы думаем, 
строить системы документооборота, причем без особых усилий. Хотя у нас еще не было 
прецедентов реализаций систем с несколькими стадиями проверок объектов на пути к полной 
публикации, простые системы с редакторами и операторами контента на LIMB уже были 
реализованы.
       На сегодняшний момент мы ввели во многие объекты только 1 стадию документооборота – 
публикацию объекта в один шаг. Мы считаем, что не нужно вводить излишнюю сложность без 
необходимости. 
       На документообороте стоит заострить внимание. Для понятия принципов документооборота 
рекомендуется понимать смысл политики доступа LIMB, в особенности шаблонов прав доступа к 
действиям. В понятиях LIMB документооборот – это просто смена статуса (значения поля) и 
изменение прав доступа к объекту. Таким образом, действие по публикации или снятия с 
публикации объекта – это всего лишь смена статуса (например, с 1 до 2), а также запись новых 
прав на объект. Скорее всего, администратор для действия публикации настроит шаблоны прав 
доступа на действия, что при публикации документ будет доступен для чтения для посетителей 
сайта, а при снятии публикации – доступ для чтения для посетителей сайта будет снят. Никакой 
логики, кроме прав доступа, при выборе объектов в общем случае не учитывается. Необходимо 
просто правильно определиться с набором действий, группами пользователей, шаблонами прав, 
доступам к действиям и правильно написать шаблоны для объектов, для которых реализуется 
особая схема документооборота.
       Такая простая схема позволяет при необходимости для некоторых объектов вводить 
сложные цепочки. Например, журналист вводит статью и делает ей статус «требуется проверка», 
статья становится доступной для записи и чтения редакторам. Редактор проверяет статью и 
помечает ее через очередное действие как «одобрена и требует публикации». Статья теперь не 
доступна журналистам для записи, но доступна главным редакторам для записи. Главный редактор 
окончательно публикует статью, при этом пользователи получают право чтения для статьи, ни 
редакторы, ни журналисты права на запись не имеют, главный редактор имеет полный доступ к 
статье. Для того, чтобы реализовать подобную схему на LIMB придется создать отдельные классы 
для работы со статьями и раздел для таких статей, несложные классы для действий по смене 
статуса, написать пару тегов для отображения текущего статуса статьи (можно воспользоваться 
уже существующими), и правильно расставить шаблоны прав доступа при выполнении 
определенных действий + доступ к самим действиям. Всю схему можно реализовать за срок от 1 
дня до 1 недели, причем будут использованы только средства LIMB.
